1. Code de l'application et documentation

Les fichiers principaux du projet sont les suivants :

  • SAE.c : Contient l'ensemble des fonctions liées au déroulement du jeu, gestion des parties, duels, affichage des scores et statistiques.
  • SAE.h : Fichier d'en-tête contenant les déclarations des structures de données et des prototypes de fonctions.
  • TestSAE.c : Contient la fonction main et un test spécifique pour le fonctionnement des duels.
  • Parties/partie1.txt : Fichier texte contenant un exemple de partie prédéfinis.
  • Parties/scoreboard.txt : Fichier texte stockant les statistiques des joueurs.

2. Fonctionnalités réalisées

  • Affichage du menu principal : Permet de choisir entre jouer, créer une partie ou afficher les scores.
  • Jouer une partie prédéfinie : Charge une partie définie dans un fichier et simule les duels.
  • Créer une nouvelle partie : Permet de créer un fichier décrivant une nouvelle partie avec des groupes de monstres.
  • Afficher le tableau des scores trié par nom : Affiche la liste des joueurs triée par ordre alphabétique.
  • Afficher le tableau des scores trié par meilleur score : Affiche les scores triés dans l'ordre décroissant.
  • Afficher les statistiques d’un joueur : Affiche le meilleur score, la moyenne des scores et le nombre de victoires/défaites.

3. Structure des fichiers utilisés

partie1.txt

  • Type : Texte
  • Structure interne :
    • Ligne 1 : Contexte du premier groupe.
    • Ligne 2 : Nombre de monstres du premier groupe.
    • Suivent les détails des monstres du groupe (nom, niveau, nombre d'armes, liste des armes).
    • Ligne après le groupe 1 : Contexte du second groupe.
    • Ligne suivante : Nombre de monstres du second groupe.
    • Suivent les détails des monstres du second groupe.
  • Exemple :
    Vous arrivez dans un corridor...
    2
    Choin 1 4 P F C O

scoreboard.txt

  • Type : Texte
  • Structure interne :
    • Ligne 1 : Nombre total de joueurs.
    • Suivent les statistiques de chaque joueur (pseudo, meilleur score, moyenne des scores, parties jouées, victoires, défaites).
  • Exemple :
    8
    DiouD 100 100 1 0 1

4. Structures de données en mémoire

  • Joueur :
    • Structure contenant le pseudo, points de vie, dégâts par attaque, tableau des scores, et nombre de parties jouées/gagnées/perdues.
  • Monstre :
    • Structure contenant le nom, points de vie, niveau, dégâts par attaque, nombre d'armes et tableau des armes.
  • Stats :
    • Contient le pseudo du joueur, meilleur score, moyenne des scores, nombre de parties, victoires et défaites.

Choix des structures :

  • Les tableaux sont dynamiques pour stocker les joueurs et les monstres, afin de s’adapter au nombre de données.
  • Les armes sont stockées dans des tableaux fixes de taille maximale MAX_ARMES pour simplifier la gestion des armes des monstres.
  • Un tableau de structures est utilisé pour les scores afin de faciliter le tri et la recherche.

5. Algorithmes de tri et recherche

Tri

  • Tri par échange :
    • Utilisé pour trier les joueurs par pseudo ou par score.
    • Complexité : O(n²), suffisante pour un nombre modéré de joueurs.

Recherche

  • Recherche dichotomique :
    • Utilisée pour rechercher un joueur par pseudo.
    • Complexité : O(log n).
    • Plus efficace qu'une recherche linéaire pour une liste triée.
